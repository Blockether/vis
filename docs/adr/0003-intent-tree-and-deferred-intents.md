# Intent tree, deferred intents, and extension intent queries

Vis will model conversation work as a single intent tree with exactly one running intent cursor per conversation. Extension-created future work starts as a suggested intent; only the user or host-owned system policy may accept it into a deferred or active commitment. Deferred intents wait on a minimal trigger vocabulary (`:defer/user-input`, `:defer/time`, `:defer/extension-signal`, `:defer/intent`), become resumable when the trigger is observed, and require a separate user or host-policy resume decision before moving the cursor. Extensions discover suggested/deferred work through durable intent queries, not an intent transition event bus, so database state remains truth and extension sidecars/nudges never become proof authority.

This keeps `PROOF.md`'s evidence boundary intact: extensions may suggest, query, nudge, and observe proof lifecycle events, but commitment transitions still flow through provenance events, evidence bundles, attestations, and audit. Subintents are real intent nodes executed depth-first by the cursor, not same-level parallel work; an intent is either decomposed into subintents or resolved by its own plan/gates, not both. Direction changes require an abandonment gate with explicit abandonment scope rather than silent deletion or cursor jumps.
